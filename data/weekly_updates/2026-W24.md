# 2026-W24（2026-06-08～2026-06-14）

## Weekly Summary

本周最重要的信号是 Claude Fable 5 / Mythos 5 的发布与随后暂停访问：模型能力上限被快速抬高到长周期 coding agent 与多日任务，但模型可用性、数据留存、安全分类器、降级和企业治理也同时变成产品能力的一部分。GitHub Copilot、Cursor 和 Claude Code 都被这个模型事件牵动；TRAE SOLO / TRAE Work 则继续从个人 SOLO 任务向企业工作入口演进。

## Head Product Signals

- **OpenAI Codex**：无同等级重大公开更新；Codex 仍应作为“CLI / app / Sites / provider / deployable artifact”路线的对照组观察。
- **Claude Code**：Claude Fable 5 作为 Mythos-class 模型发布，明确面向 Claude Code / Managed Agents 这类 agent harness，可做多日长任务、分阶段规划、子代理委派和自测；但 6 月 12 日 Fable 5 / Mythos 5 访问被暂停，模型可用性与治理风险成为 coding agent 产品必须处理的第一等问题。
- **Cursor**：Cursor Docs 已将 Claude Fable 5 定位为面向 autonomous long-running agentic work 的模型，Privacy Mode 用户需要单独接受 Anthropic 数据留存；CursorBench 3.1 中 Fable 5 最高 72.9%，但成本也显著更高，说明 Cursor 的产品竞争正在进入“模型选择 + 成本 / 隐私解释 + agent benchmark”阶段。
- **TRAE SOLO**：TRAE SOLO 在本周升级为 TRAE Work，并补企业版账号登录、登录设备管理与模型选择；这意味着 SOLO 叙事从个人 agent 体验继续转向团队和企业入口。
- **GitHub Copilot**：GitHub Copilot 在 6 月 9 日开放 Claude Fable 5，并强调其 long-horizon autonomous coding / knowledge-work 定位、数据留存要求和 usage-based billing；同周还发布 Agentic Workflows public preview、Copilot Chat sees agent sessions、第三方 coding agent 安全校验，路线是把多模型 agent 纳入 GitHub 原生治理和自动化体系。
- **Windsurf / Devin Desktop**：无同等级重大公开更新；继续观察 Devin Desktop 是否把模型升级、agent 工作台和本地 / 云端迁移做成稳定交付链路。
- **OpenClaw**：本周延续 channel 输出净化、MCP rich-content、插件状态和 runtime recovery 的安全硬化方向；开源 agent runtime 的关键不只是接入更多模型，而是把多渠道交付和工具结果边界做稳。
- **Kimi Code**：本周继续观察 agent swarm、Claude Code / Codex 迁移、插件管理和配置能力；对国内产品的关键问题是能否把较弱或更便宜模型通过产品层包装成可恢复的长任务 agent。
- **Zhipu GLM Coding Plan / CodeGeeX**：无

## Competitive Reading

- **路线判断**：Fable 5 事件说明 AI coding 的竞争不再只是“谁模型更强”，而是“强模型突然上线 / 下线 / 有数据留存限制时，产品是否还能保持任务连续、成本可控、权限可解释”。
- **模型能力变化**：Fable 5 把长周期 agent coding 的能力上限提高，但其 1M context、128k output、$10 / $50 per million tokens、30 天数据留存和安全 fallback，都会直接影响产品的模型选择器、套餐、预算和企业合规。
- **产品能力补位**：CursorBench、GitHub Copilot 模型策略、Claude API fallback credit 共同说明：模型越强，产品越要补模型生命周期提示、隐私/留存确认、fallback、成本预估、任务暂停恢复和验收审查。

## TRAE SOLO Implication

TRAE SOLO / TRAE Work 不能只问“接哪个模型最强”，而要把模型可用性、模型成本、数据留存、敏感任务降级、任务恢复和企业权限解释成用户能理解的业务状态。面向非工程师时，用户不应该看到一堆模型异常，而应该看到：当前任务是否还能继续、换模型会损失什么、成本会增加多少、哪些资料不能交给模型、交付物是否仍可验收。

## LPME Implication

LPME 需要新增“模型生命周期与供应风险”评分项：模型上线 / 下线 / 暂停访问时，产品是否明确提示、是否支持降级、是否保留任务上下文、是否解释成本和隐私变化、是否能重新验证交付物。还应新增“长任务强模型性价比”记录：完成率、token / credit 消耗、人工接管次数、失败恢复时间和最终可验收程度。
